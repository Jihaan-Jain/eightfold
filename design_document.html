Problem Statement
Recruiter tooling produces conflicting, duplicate candidate records
across CSV exports, ATS blobs, résumés, and GitHub. This system
resolves identity and emits one canonical, confidence-scored profile.

Key Design Decisions
CandidateProfile frozen; output schema separate
Projection reshapes without mutating the canonical record
Email is a hard signal — no weight accumulation
Emails are globally unique; one exact match is provably sufficient
SBERT scoped strictly to skill synonym resolution
Keeps merge path fully deterministic; non-reproducible embeddings isolated
FieldRegistry as single canonical field definition
All 4 mappers share one registry; new field = one-line change
Rule-based conflict resolution over learned ranking
Every merge decision is inspectable, reproducible, and debuggable

Scalability Path
Identity resolution is O(n²) — the sole bottleneck.
→ Email-domain blocking cuts comparisons 10–100×
→ All other stages are O(n), trivially parallelisable
→ Stateless stages → queue / worker sharding ready
→ SBERT lazy-loaded; GPU-acceleratable on demand
